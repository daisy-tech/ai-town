# 疯狂动物城 PRD 评审与架构决策

> 项目代号 AI-PetTown。本文档记录初版 PRD（AI Pet Town MVP v0.1）的评审结论，
> 以及"为什么基于现有 AI-Town 代码库实现、而不是按 PRD 重写"的技术决策依据。

## 1. 初版 PRD 回顾

初版 PRD 提出的 MVP 范围：

- 一张 100m×100m 地图（森林、小屋、花园、河边、果树）。
- 5 只 AI 宠物（狐狸、猫、熊猫、柴犬、兔子），各有固定性格、动态情绪、长期记忆、社交关系和每日日程。
- 玩家能力：行走、点击宠物、聊天、送礼、查看日记。
- Agent Loop：观察世界 → 决定目标 → 移动 → 互动 → 更新记忆。
- 技术架构：**Unity + REST/WebSocket + FastAPI + Postgres + Redis + VectorDB**。
- 四个 Sprint 的开发里程碑。

## 2. 评审结论：产品方向可行，架构方案替换

### 核心判断

PRD 的产品目标（生命感、每日回访、持续故事）与现有 AI-Town 代码库的能力**高度重合**。按 PRD 的 Unity/FastAPI 架构重写，等于放弃一套已经跑通的系统，重新踩一遍坑；正确做法是在现有架构上做主题化和增量开发。

### 现有代码库已具备的能力（直接复用）

| PRD 需求 | AI-Town 现成实现 |
| --- | --- |
| 可探索的小镇地图 | PixiJS 瓦片地图渲染 + 碰撞 + 寻路 |
| 宠物自主移动 | 单线程游戏引擎，输入队列 + 定时 tick |
| 宠物之间自动聊天 | Agent 自主发起/接受对话，LLM 生成内容 |
| 玩家与宠物聊天 | 已有玩家加入世界、点击角色对话的完整链路 |
| 长期记忆 | Convex 向量记忆：对话总结、重要性打分、反思、向量检索 |
| 关系随互动变化 | 记忆中的关系事实抽取与召回 |
| 玩家离线世界继续模拟 | 后端引擎独立于前端运行 |

### 需要新建的能力（增量开发）

- 专属宠物绑定（1 孩子 ↔ 1 宠物）与宠物创建流程。
- 每日日程/事件池（学堂事件、成长任务、公共事件）。
- 情绪状态机（规则驱动，影响表达与行为）。
- 日记系统（每日自动生成、玩家可查看）。
- 送礼系统。
- 秘密花园模式（四种回应方式 + 记忆控制）。
- 儿童安全层（年龄分层、三层回应、禁止表达、求助引导）。

### 架构对比决策表

| 维度 | PRD 方案（Unity+FastAPI） | 采用方案（现有架构） |
| --- | --- | --- |
| 前端 | Unity 客户端，需要新建整个渲染/交互层 | React + PixiJS，已有完整地图与角色渲染 |
| 后端 | FastAPI + Postgres + Redis + VectorDB 四件套 | Convex 自托管（数据库/函数/向量索引一体） |
| Agent Loop | 从零实现 | 复用现有 Agent 操作与调度 |
| 记忆系统 | 自建 VectorDB 接入 | 复用 Convex 向量索引与记忆管线 |
| 交付周期 | 4 个 Sprint 全部花在重建基础设施 | 基础设施为零成本，直接做产品差异化 |
| 运维 | 多组件部署 | 单 docker compose，已在阿里云 ECS 跑通 |

## 3. 落地实现记录（截至 2026-07-12）

- 新分支 `ai-pet-town`，主题从"奥特曼光之国"整体切换为"疯狂动物城"。
- 80×60 新地图（`data/petTownGraybox.ts`）：心愿树广场、共享之家、成长学堂与操场、果园花园、图书兴趣馆、生活街、河边与安静草地；先灰盒验证寻路，再分区域接入正式像素美术。
- 五只宠物定稿：Mochi狐、Nana猫、Bobo熊猫、刀盾狗（替代柴犬 Coco）、Mimi兔；行走图与头像由脚本从定板自动生成。
- 活动与提示词全部宠物主题化（`convex/constants.ts`、`convex/agent/memory.ts`）。
- 深度排序（sortY）渲染机制，让角色能从大型装饰前后走过。
- 预览环境：独立 docker compose 项目（后端 3310 端口），与线上 ECS 环境隔离。
